# 《计算机专业简历编写完全指南》

## 一、写在前面：为什么你的简历总是没回应？

### 风格：简洁克制，比“好看”更重要

很多同学会在网上找一些所谓的“高级简历模板”，页面做得花里胡哨，配色复杂，图标密集，看起来仿佛很“设计感”。

但现实是什么？

大多数简历最终都是：

* 黑白打印。
* A4 纸。
* 快速浏览。
* 扫描式阅读。

那些你精心挑选的颜色、渐变、图表、进度条，打印出来只会变成一团灰黑色块，不仅没有加分，反而显得混乱、廉价、难看。

技术岗简历的核心不是视觉设计，而是信息表达效率。

一份好的简历应该做到：

* 黑白打印依然清晰。
* 没有多余装饰。
* 不依赖颜色区分信息。
* 信息层级靠排版，而不是靠花哨设计。

记住一句话：简历不是作品集封面，它是信息密度极高的技术文档。

### 格式：卷面差，直接出局

很多同学的简历存在一个严重问题 —— 版式混乱。

常见问题包括：

* 字体不统一（宋体、微软雅黑、黑体混着来）。
* 字号忽大忽小。
* 行间距忽紧忽松。
* 标题格式不统一。
* 对齐方式混乱。
* 加粗乱用。
* 标点符号中英文混乱。

看一份这样的简历是什么感觉？就像高中老师批作文一样 —— 卷面不干净，直接扣印象分。

但找工作不是扣分制，而是淘汰制，HR 或面试官在筛选简历时，停留时间可能只有 10 秒。如果排版让人阅读成本很高，他们不会费力理解你，而是直接关闭文档。

一份专业的简历，至少应该做到：

* 字体统一。
* 字号分级清晰。
* 标题结构一致。
* 行距统一。
* 所有内容对齐规范。
* 整体视觉平衡。

简历首先要“让人愿意看”，其次才是“看得懂”，最后才是“觉得你不错”。

如果前两步都没做到，后面根本没有机会。

### 结构权重：重要的内容，要放在重要的位置

很多同学在写简历时没有“信息权重”概念。

常见错误包括：

* 把自我评价写在最前面。
* 把项目经历放在后面。
* 技术栈写得很简略。
* 花大量篇幅写无关内容。
* 写一堆不痛不痒的经历。
* 真正有含金量的东西却写得很靠后。

简历不是流水账，简历是一份“营销文档”，每一行内容都在竞争注意力。核心原则只有一句话：重要信息靠前，不重要信息靠后。

对于计算机专业学生来说，常见模块有：

- 项目经历。
- 实习经历。
- 技术栈。
- 教育背景。
- 比赛 / 荣誉 / 其他。

**谁更重要谁就放在最前面** —— 根据你的实际情况，把最能体现你优势的模块排到前面，让面试官最先看到。如果顺序错了，哪怕你内容很好，也会被埋没。面试官不会帮你挖亮点，你必须把亮点主动摆在他视线最先落下的地方。

> 谁重要一定要自我权衡好！学历是北京大学，那教育经历就放在最前面；参加过很牛逼的大厂实习，实习经历肯定就在前面。自我权衡！！一定要将最最最重要的东西放在最前面！！

### 内容问题：“空” 和 “水”

#### “空” —— 没有具体描述，没有数据，没有技术深度，没有成果。

典型特征：

* **只写技术名词**：技术栈一栏堆满 Spring、Redis、Kafka，但项目里完全看不出这些技术是怎么用的、解决了什么问题。
* **写“参与开发”**：只说“参与 XX 系统开发”，没说负责哪块、做了什么、用了什么方案，等于什么都没说。
* **写“熟悉某某技术”**：自称熟悉，却没有项目或经历来佐证，面试官无从判断真假。
* **没有具体描述**：只有岗位名称和时间，没有职责、没有动作、没有产出，像在填表格。
* **没有数据**：没有用户量、没有 QPS、没有优化比例、没有业务指标，无法量化你的贡献。
* **没有技术深度**：只写“实现了 XX 功能”，不写实现思路、技术选型、遇到的难点和解决方案。

**一句话概括**：空 = 信息量为零。看完之后，面试官不知道你具体做过什么、会什么、能带来什么价值。

这样的简历看完之后，没有记忆点，面试官无法判断你具体的能力。

#### “水” —— 描述大量无技术含量的内容，字数很多，但没有技术亮点。

典型特征：

* **描述大量无技术含量的内容**：大段写需求背景、项目介绍、团队分工，唯独不说自己做了什么、怎么做的。
* **叙述过程而不是成果**：像写记叙文一样流水账 —— “先做了 A，然后做了 B，最后做了 C” —— 面试官关心的是你解决了什么问题、带来了什么结果，不是你的工作流水。
* **字数很多，但价值很低**：一段话占半页，拆开看全是“参与”、“协助”、“了解”，没有一句能体现你的技术贡献。

**核心原则**：精准描述自己的贡献，而不是长篇大论像写记叙文。简历不是日记，不需要记录你每天在干嘛；每一行都要回答 —— **这件事里，你具体做了什么、用了什么技术、带来了什么效果**。

简历的本质不是记录经历，而是展示能力。你要回答的是：

* 你解决过什么问题？
* 你做过什么有技术含量的事情？
* 你优化过什么？
* 你理解到什么层面？
* 你带来了什么结果？

如果内容不能体现能力，再多文字也没有意义。

## 二、简历的整体结构设计

简历最少两页，最多三页。太短显得单薄，一页塞满密密麻麻的小字，阅读体验很差，面试官看着也累；太长则没完没了，重点被稀释，没人有耐心翻到第五页。根据你的经历量来调整：经历多就两到三页从容展开，经历少就两页把该说的说清楚，不要硬挤也不要硬凑。

> 不管有几页，都最好把每一页刚好写满。可以通过调整内容和行距来做到，这样可以让你的简历看起来不那么空荡荡的。

## 三、基础信息该怎么写？

### 必须包含的内容

以下是必须包含的内容：

* **姓名**。
* **性别**。
* **学历**（写清本科 / 专科 / 研究生即可，无需在此处展开学校介绍）。
* **年龄**。
* **联系电话**。
* **邮箱**。

> 关于邮箱：国内企业使用 QQ 邮箱是可以接受的；如果投递外企或偏国际化团队，建议使用 Gmail 等更通用邮箱，更显专业。

### 不建议写的内容

以下内容通常没有必要写在技术岗位简历中：

* 身高、体重。
* 民族、政治面貌（除非特定岗位或明确要求）。

技术岗位更关注能力，而不是个人外在标签，无关信息越多，反而越干扰重点内容的表达。

### 是否需要放照片？

关于照片，其实是一个策略问题。如果有比较帅的证件照，直接搞上去，非常加分。主播也是小帅的好吧，本身就是有优势的。如果是那种又丑又垃圾的照片，就别搞上去了，纯减分。如果没有照片，花点钱去拍一张好的，稍微 P 得好看一点，别扣扣搜搜地心疼那点钱。

## 四、教育背景写法

### 排序方式（时间倒序）

按照时间倒序来写，从你**最近的学历**往下写，也就是博士、硕士、本科这样的顺序。一般写到本科即可，高中没有必要再写上去。

### 是否写 GPA？

GPA 如果比较高，直接写，不要犹豫；如果 GPA 比较低，那就别写了。有没有挂科也不要写，写出来基本就是给自己减分，没必要。

### 相关课程要不要列？

可以列几门和专业匹配度最高的核心课程，比如：C++、Java、计算机网络、操作系统原理、数据结构与算法、计算机组成原理等。

这里可以适当权衡选择，不用全写，挑对岗位有帮助的几门即可。如果是大学期间系统自学过的相关课程，只要确实掌握，也可以列出来，不必过于死板。

## 五、技术栈怎么写才不“空”？

> 这一部分是重点。

先来看一个错误示例：“熟悉 Java、MySQL”。

你这里直接把几个技术名词罗列在一行里，这是在干嘛？本来可以展开成两行有信息量的内容，你直接压缩成一行，信息密度直接归零。

而且，每一个技术栈体量都很大，你真的全部都“熟悉”吗？

正确做法是：

* 每个技术栈单独占一行。
* 后面补充你**具体掌握了什么**。
* 写清楚理解到什么程度

这样一来，面试官就可以从你补充的内容作为切入点来提问，而不是随便挑一个他熟、你未必熟的角度开始拷打。

来看改进示例：

* 熟练掌握 Java 基础知识，如集合、反射、IO 等，能够使用 Java 语言独立完成业务代码的编写。
* 熟悉 MySQL 数据库，对事务、索引、MVCC 等机制有一定理解，能够编写常用 SQL 语句，并进行简单的 SQL 优化。

这种写法至少体现了三个东西：

1. 你确实知道自己会什么。
2. 你对技术有拆解能力。
3. 面试官能根据你写的点深入提问。

补充一点 —— **避免“精通”陷阱**。

不要轻易写“精通”，真的不要。计算机是一个需要长期积累的学科，大多数人都处在不断学习的过程中。你敢写“精通”，面试官很可能就按“精通”的标准来考你。写了“精通”，基本等于给自己提高难度，如果撑不住，就是给自己挖坑。

## 六、项目经历 —— 简历最核心部分（重中之重）

首先打个预防针 —— **如果你写的项目不是你在公司真实参与过的实习项目，千万不要硬说成是实习项目**。真的，一眼就能看出来，面试官要是稍微较真一点，顺着细节往下问，你基本当场就露馅。

所以，对于自己独立完成的项目，想写进简历里，只有两个思路：

* 要么大方承认这是一个学习项目，主要用于练手、巩固和实践你学过的技术。
* 要么换一个合理的角度进行包装，比如这是学院的实际需求项目，或者是学校内部使用的小系统，是你和几位同学一起完成的。

关键是一定要逻辑自洽、前后说得通，只要细节对不上，一旦被追问，你就会非常被动。

举个例子 —— 假如你写的是 B 站上一个非常经典的外卖项目，那你完全可以换个合理的背景去表达，比如说：这是学校内部为了搭建校园外卖平台而做的项目，你和几位同学一起完成的，逻辑说得通，场景自洽，就可以。

对于应届生来说，说句现实一点的话 —— **90% 的应届生本来就没有做过什么特别牛的商业级项目**，面试官其实心里清楚这一点。

面试的重点从来都不是项目有多“震撼”，而是：

* 你在项目里做了什么。
* 你掌握了哪些技术。
* 你理解到了什么深度。
* 你能不能围绕技术展开讨论。

不要被网上卖课的机构忽悠了，什么“项目必须高并发”、“必须高可用”、“必须微服务架构”、“必须分布式”、“必须日活几十万”。很多人工作进公司以后，写的就是正常的增删改查。一些复杂架构的东西，可能工作几年之后才有机会真正深入接触。那些机构强调项目必须“看起来很炸”，本质上很多时候只是为了制造焦虑，让你觉得自己不够好，从而去报他们的课。

项目不用追求浮夸，能讲清楚技术原理、设计思路和取舍逻辑，比堆一堆听起来很厉害的名词，更有价值。

现在回归到项目经历本身 —— 怎么写才能让面试官感兴趣？

在编写项目经历时，结构一定要清晰、完整。一般来说，你需要明确写清以下几个部分：

* **项目名称**。
* **开发时间段**。
* **项目描述**。
* **主要技术**。
* **主要职责**。

项目名称尽量自己取一个合理、正式一点的名称。不要把 B 站教程里的名字直接原封不动写上去，什么“黑马外卖”、“瑞吉外卖”、“苍穹外卖”之类的直接照搬，真的很尴尬。一看就是教程项目，显得很不认真。

开发时间段也要写得合理，根据项目体量来定：

* 项目规模较大，可以写几个月；
* 项目规模较小，写几周即可。

时间一定要符合常识，不要一个简单的小项目写成半年，也不要一个看起来很复杂的系统只写两周。

项目描述的作用是交代背景和目标，说清楚这个项目是做什么的、解决什么问题。
这一部分保持简洁即可，短小精悍，不要在这里堆字数。重点不是写故事，而是快速让人理解项目场景。

主要技术这一部分，就是写清楚你在这个项目中实际用到了哪些技术。也就是把项目涉及到的技术栈列出来，比如：Spring、SpringMVC、SpringBoot、MyBatis、Redis、Lombok、RabbitMQ、MySQL、Docker 等。需要注意的是，这里写的一定是你真实使用过、并且能够讲清楚原理和用法的技术，而不是单纯为了显得丰富而堆名词。面试时，这一栏基本都会被重点提问。

主要职责这一部分，就是写清楚你在这个项目中具体负责什么。包括：

* 你负责了哪些模块。
* 实现了哪些功能。
* 承担了哪些核心任务。

这一部分是整个项目经历里**最核心的内容**，一定要分条分点写清楚。

重点说明：

* 你具体做了什么。
* 解决了什么问题。
* 遇到了哪些困难。
* 是如何分析并解决的。
* 最终带来了什么结果或提升。

不要笼统地写“负责开发某模块”，而是要拆开讲清楚你的实际贡献。

这一部分写得好不好，基本决定了面试官会不会愿意围绕你的项目深入提问。
