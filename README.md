# 《计算机专业简历编写完全指南》

## 一、写在前面：为什么你的简历总是没回应？

### 风格：简洁克制，比“好看”更重要

很多同学会在网上找一些所谓的“高级简历模板”，页面做得花里胡哨，配色复杂，图标密集，看起来仿佛很“设计感”。

但现实是什么？

大多数简历最终都是：

* 黑白打印。
* A4 纸。
* 快速浏览。
* 扫描式阅读。

那些你精心挑选的颜色、渐变、图表、进度条，打印出来只会变成一团灰黑色块，不仅没有加分，反而显得混乱、廉价、难看。

技术岗简历的核心不是视觉设计，而是信息表达效率。

一份好的简历应该做到：

* 黑白打印依然清晰。
* 没有多余装饰。
* 不依赖颜色区分信息。
* 信息层级靠排版，而不是靠花哨设计。

记住一句话：简历不是作品集封面，它是信息密度极高的技术文档。

### 格式：卷面差，直接出局

很多同学的简历存在一个严重问题 —— 版式混乱。

常见问题包括：

* 字体不统一（宋体、微软雅黑、黑体混着来）。
* 字号忽大忽小。
* 行间距忽紧忽松。
* 标题格式不统一。
* 对齐方式混乱。
* 加粗乱用。
* 标点符号中英文混乱。

看一份这样的简历是什么感觉？就像高中老师批作文一样 —— 卷面不干净，直接扣印象分。

但找工作不是扣分制，而是淘汰制，HR 或面试官在筛选简历时，停留时间可能只有 10 秒。如果排版让人阅读成本很高，他们不会费力理解你，而是直接关闭文档。

一份专业的简历，至少应该做到：

* 字体统一。
* 字号分级清晰。
* 标题结构一致。
* 行距统一。
* 所有内容对齐规范。
* 整体视觉平衡。

简历首先要“让人愿意看”，其次才是“看得懂”，最后才是“觉得你不错”。

如果前两步都没做到，后面根本没有机会。

### 结构权重：重要的内容，要放在重要的位置

很多同学在写简历时没有“信息权重”概念。

常见错误包括：

* 把自我评价写在最前面。
* 把项目经历放在后面。
* 技术栈写得很简略。
* 花大量篇幅写无关内容。
* 写一堆不痛不痒的经历。
* 真正有含金量的东西却写得很靠后。

简历不是流水账，简历是一份“营销文档”，每一行内容都在竞争注意力。核心原则只有一句话：重要信息靠前，不重要信息靠后。

对于计算机专业学生来说，常见模块有：

- 项目经历。
- 实习经历。
- 技术栈。
- 教育背景。
- 比赛 / 荣誉 / 其他。

**谁更重要谁就放在最前面** —— 根据你的实际情况，把最能体现你优势的模块排到前面，让面试官最先看到。如果顺序错了，哪怕你内容很好，也会被埋没。面试官不会帮你挖亮点，你必须把亮点主动摆在他视线最先落下的地方。

> 谁重要一定要自我权衡好！学历是北京大学，那教育经历就放在最前面；参加过很牛逼的大厂实习，实习经历肯定就在前面。自我权衡！！一定要将最最最重要的东西放在最前面！！

### 内容问题：写了很多，依旧 “空” 和 “水”

#### “空” —— 没有具体描述，没有数据，没有技术深度，没有成果。

典型特征：

* **只写技术名词**：技术栈一栏堆满 Spring、Redis、Kafka，但项目里完全看不出这些技术是怎么用的、解决了什么问题。
* **写“参与开发”**：只说“参与 XX 系统开发”，没说负责哪块、做了什么、用了什么方案，等于什么都没说。
* **写“熟悉某某技术”**：自称熟悉，却没有项目或经历来佐证，面试官无从判断真假。
* **没有具体描述**：只有岗位名称和时间，没有职责、没有动作、没有产出，像在填表格。
* **没有数据**：没有用户量、没有 QPS、没有优化比例、没有业务指标，无法量化你的贡献。
* **没有技术深度**：只写“实现了 XX 功能”，不写实现思路、技术选型、遇到的难点和解决方案。

**一句话概括**：空 = 信息量为零。看完之后，面试官不知道你具体做过什么、会什么、能带来什么价值。

这样的简历看完之后，没有记忆点，面试官无法判断你具体的能力。

#### “水” —— 描述大量无技术含量的内容，字数很多，但没有技术亮点。

典型特征：

* **描述大量无技术含量的内容**：大段写需求背景、项目介绍、团队分工，唯独不说自己做了什么、怎么做的。
* **叙述过程而不是成果**：像写记叙文一样流水账 —— “先做了 A，然后做了 B，最后做了 C” —— 面试官关心的是你解决了什么问题、带来了什么结果，不是你的工作流水。
* **字数很多，但价值很低**：一段话占半页，拆开看全是“参与”、“协助”、“了解”，没有一句能体现你的技术贡献。

**核心原则**：精准描述自己的贡献，而不是长篇大论像写记叙文。简历不是日记，不需要记录你每天在干嘛；每一行都要回答 —— **这件事里，你具体做了什么、用了什么技术、带来了什么效果**。

简历的本质不是记录经历，而是展示能力。你要回答的是：

* 你解决过什么问题？
* 你做过什么有技术含量的事情？
* 你优化过什么？
* 你理解到什么层面？
* 你带来了什么结果？

如果内容不能体现能力，再多文字也没有意义。

## 五、技术栈怎么写才不“空”？

> 这一部分是重点。

1. 技术栈的正确写法（不要只写名词）
2. 错误示例：

   * “熟悉 Java、MySQL、Redis”
3. 改进示例：

   * 有具体能力说明
   * 有项目支撑
4. 如何分层：

   * 编程语言
   * 框架
   * 数据库
   * 中间件
   * 工程化工具
5. 避免“精通”陷阱

## 六、项目经历 —— 简历最核心部分（重中之重）

1. 项目选择原则

   * 不要堆数量
   * 选最有技术含量的
2. 标准写作结构：

   * 项目背景
   * 技术架构
   * 个人职责
   * 技术亮点
   * 数据成果（量化）
3. STAR 法则在项目中的运用
4. 如何写技术深度？

   * 性能优化
   * 架构设计
   * 解决复杂问题
5. 错误示例与改写对比
6. 不要写：

   * “负责页面开发”
   * “参与系统维护”
